# Issue #46: Fiscal Year Cutover Strategy

## TL;DR
When a new FY starts, archive the current database, run MSX account sync to overlay new alignments, continue working while alignments churn, then finalize by purging orphaned accounts. A read-only archive browser lets you reference previous years' notes without affecting the live database.

## The Problem

Every fiscal year:
1. Sellers get realigned to new territories, customers, PODs
2. Customer TPIDs persist (same companies, different ownership)
3. Some customers/sellers may carry over, others won't
4. First month is chaos — alignments change 3-4x before finalizing
5. Valuable data (notes, engagements) should persist for retained customers

## Architecture: "Overlay + Purge"

```
data/
├── notehelper.db              # Active database (current FY)
├── FY25.db                    # Local archive (always created)
├── FY24.db                    # Older archives

OneDrive/backups/notehelper/
├── previous_years/
│   ├── FY25.db                # OneDrive archive (if backups enabled)
│   └── FY24.db
```

**Key Insight:** Don't create a new database — overlay new alignments on the existing one, then purge what's no longer relevant. Notes naturally follow their customers (via TPID).

## Workflow

### Step 1: "Start New Fiscal Year" (Admin Panel)
**When:** New FY begins, user has tentative alignments in MSX

**What happens:**
1. Create archive backup of current database:
   - **Always:** `data/FY25.db` (local copy)
   - **If OneDrive enabled:** Also copy to `OneDrive/backups/notehelper/previous_years/FY25.db`
2. Show success message with archive location
3. Set app into "FY Transition Mode" (banner: "FY26 Setup — Alignments not finalized")

### Step 2: MSX Account Sync (New Alignments)
**What happens:**
1. User clicks "Sync Accounts from MSX" (existing feature)
2. Creates new PODs (they change annually)
3. Creates new territories/sellers as needed
4. For existing customers (TPID match): updates seller/territory assignment
5. For new customers: creates them fresh
6. **Result:** Full previous FY data + new org structure overlaid

### Step 3: Holding Pattern (Work Continues)
**Duration:** ~3-4 weeks while alignments stabilize

- All old notes and engagements still accessible
- New sellers/PODs/territories in place
- User keeps taking notes, creating engagements normally
- Can re-run MSX sync as alignments change in MSX
- Banner reminds user alignments aren't finalized

### Step 4: "Finalize Alignments" (Admin Panel)
**When:** User confirms accounts are stable (~late July)

**What happens:**
1. Run final MSX account sync
2. Get list of all account TPIDs from sync
3. **Purge orphaned data:**
   - Customers NOT in finalized TPID list → deleted
   - Cascade delete: their notes, engagements, milestones, MSX tasks, revenue data
   - Sellers with 0 remaining customers → deleted
   - Territories with 0 remaining customers → deleted  
   - PODs with 0 remaining territories → deleted
4. If OneDrive backups enabled:
   - Purge incremental `.json` note backups
   - Run full backup resync so only current notes are backed up
5. Remove "FY Transition Mode" banner
6. Show summary: "Kept X customers, Y notes. Purged Z orphaned accounts."

### Step 5: Normal Operations
- FY26 fully operational with clean data
- FY25 archive available for reference
- Only relevant accounts + their full history remain

## Archive Browser (Read-Only)

### Access Point
Admin Panel → "Previous Fiscal Years" section shows available archives

### UI Flow
```
Admin Panel
    └── Previous Fiscal Years
            ├── FY25.db (142 MB) — Archived Jul 1, 2025 → [Browse Archive]
            └── FY24.db (98 MB) — Archived Jul 1, 2024 → [Browse Archive]

Archive Browser (/archive/FY25)
    ├── Search bar (search notes/customers within archive)
    ├── Customer list with note counts
    │       └── Click customer → View their notes + engagements
    └── [← Back to Live App] button
```

### Features
- **Read-only** — No edit/delete buttons, display only
- **Search** — Find notes by content, customer name
- **Customer drill-down** — See all notes and engagements for a customer
- **Visual distinction** — Different header color, "Archive Mode (Read-Only)" banner
- **Separate URL space** — `/archive/FY25/customers`, `/archive/FY25/notes/123`

### Technical Approach
Each page request:
1. Opens temporary read-only SQLite connection to archive file
2. Runs query, renders page
3. Closes connection
4. Main database stays connected for normal operations

## Cascade Delete Details

When purging a customer, these related records must be deleted:

| Table | Foreign Key | Cascade |
|-------|-------------|---------|
| `notes` | `customer_id` | Delete all notes for customer |
| `engagements` | `customer_id` | Delete all engagements |
| `milestones` | `customer_id` | Delete all milestones |
| `msx_tasks` | `milestone_id` → `customer_id` | Delete via milestone cascade |
| `customer_revenue_data` | `customer_id` | Delete revenue records |
| `product_revenue_data` | `customer_id` | Delete product revenue |
| `revenue_analysis` | `customer_id` | Delete analysis records |
| `revenue_engagements` | `analysis_id` → `customer_id` | Delete via analysis cascade |
| `opportunities` | `customer_id` | Delete opportunities |

After customer purge, clean up empty org entities:
1. Query sellers with 0 customers → delete
2. Query territories with 0 customers → delete  
3. Query PODs with 0 territories → delete

## OneDrive Backup Integration

### Archive Copy
```python
def copy_archive_to_onedrive(local_path: str, fy_label: str) -> str | None:
    """Copy archive to OneDrive if backups enabled."""
    prefs = get_user_preferences()
    if not prefs.backup_enabled or not prefs.onedrive_path:
        return None
    
    dest_dir = Path(prefs.onedrive_path) / "previous_years"
    dest_dir.mkdir(exist_ok=True)
    dest_path = dest_dir / f"{fy_label}.db"
    shutil.copy2(local_path, dest_path)
    return str(dest_path)
```

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Archive storage | Local always + OneDrive if enabled | Redundancy, user choice |
| DB approach | Overlay + purge (not fresh DB) | Preserves notes for retained customers automatically |
| Orphan handling | Hard delete with cascade | Clean slate, archive has full history if needed |
| Archive access | Read-only browser | Simple, covers 95% of use case |
| Empty org entities | Delete | No value keeping ghost sellers/territories |
| Revenue data | Purge with customer | FY-specific, fresh import each year |

---

## Out of Scope (For Now)
- "Exclude from purge" checkbox for specific customers
- Full read/write access to archives (would require server restart)
- Automated FY detection (manual trigger is fine)
- Cross-archive search (search one archive at a time)
- Archive retention policy / auto-delete old archives
